## 1. 这套文档如何使用

这套文档把原始构想扩写成可以拆分给人类或 AI 逐项实施的“制作基线”，不是一次性写死的最终答案。开发过程中，任何规则变化都应先修改对应文档，再修改数据和代码，避免策划、实现和测试各自持有不同版本。

发生冲突时，按以下优先级判断：

1. 本文档中的“已冻结决策”；
2. 各专项策划中的明确规则、公式和验收条件；
3. 内容清单与示例数据；
4. 原始构想文档（若工作区仍保留）；
5. 临时讨论、聊天记录和未归档的 AI 输出。

文档中的标签含义：

- **原型**：只为验证玩法是否成立，可以使用占位素材和简化数据；
- **MVP**：第一份可以从开始玩到结局、可交给外部玩家测试的完整版本；
- **正式版**：MVP 验证成功后才扩展的目标；
- **暂定**：允许替换，但替换时必须同步检查所有引用；
- **后置**：当前不开发，仅保留接口或设计方向。

## 3. 已冻结的设计决策

“冻结”表示后续文档以此为基线，并不代表永远不能改。若要修改，必须记录理由和受影响模块。

| 项目 | 当前决策 | 理由 |
|---|---|---|
| 暂定名称 | 《末日坐标：Aftermap》 | 保留原构想，中文强调现实地点，英文便于识别 |
| 核心类型 | 单机、2D 像素、生存管理、战术探索、叙事模拟 | 集中资源在系统互动和角色故事，而非动作表现 |
| 首发平台 | Windows PC，键鼠优先 | 最适合开发、调试和数据驱动内容生产 |
| 商业形态 | 买断制；MVP 不设计内购、联网竞争和日常任务 | 避免运营系统破坏永久死亡与叙事节奏 |
| 玩家身份 | 避难所的决策者，不强制对应某个“天选主角” | 任何角色都可能死亡，群体才是战役主角 |
| 视角 | 城市层为真实地图抽象；探索层为俯视偏斜网格 | 兼顾地点辨识度与低成本战术场景 |
| 战斗 | 半自动即时战斗，可随时暂停下达指令 | 强调准备、取舍和撤退，不强调手速 |
| 核心队伍 | 初始 4 人；探索队最多 4 人 | 保证每个角色可辨认，控制操作复杂度 |
| 人口上限 | MVP 12 人；正式版软上限 20 人 | MVP 先验证关系密度和基地岗位循环 |
| MVP 战役 | 30 个游戏日；完成一次“离开本城”的迁徙 | 给资源压力清晰终点，同时预告全球旅程 |
| MVP 地图 | 精修 1 个预设城市；另有 2 个结构测试区域 | 底层支持任意地点，不让内容质量被地图规模稀释 |
| 任意城市 | MVP 中为实验功能，生成失败时提供预设地图回退 | 现实数据质量不可控，必须保证游戏始终可玩 |
| 地图结构 | 世界航线层、城市行动层、战术探索层 | 不制作无缝全球开放世界 |
| 失败规则 | 社区无人存活为彻底失败；基地失守、错过迁徙可形成坏结局 | 保留失败故事，不因单人死亡立即结束 |
| 存档 | 默认普通存档；“仅自动存档”的生存模式后置 | 测试阶段必须方便复现，正式版再强化压力 |
| 推荐引擎 | Godot 4.x + GDScript；地图预处理工具可用 Python | 2D 能力成熟、文本资源友好、适合 AI 辅助和小团队 |
| 联网边界 | 核心游玩可离线；只有首次地点生成可能访问地图服务 | 避免地图接口故障导致无法继续已有战役 |

## 4. 三条设计支柱

### 4.1 熟悉的地方变成求生问题

现实地图不能只是背景皮肤。道路、河流、建筑密度和设施分布必须改变路线、风险与资源结构。玩家看到熟悉地点时，应当能产生“我知道这里，但末日改变了它”的判断。

检验问题：换成随机矩形地图后，这次决策是否仍完全相同？如果是，现实地图还没有真正参与玩法。

### 4.2 幸存者是有记忆的人，不是劳动力卡牌

角色会因职业、性格、关系和经历提出意见、拒绝命令、互相保护或离开。永久死亡带走的是一段关系网和未完成的故事，而不只是数值。

检验问题：玩家能否不看姓名就把两个同技能角色随意互换？如果能，人物系统还不够具体。

### 4.3 每次离开都有代价，每座城终将被留在身后

搜刮、救人、战斗和迁徙都不能获得完美答案。基地提供安全感，但资源枯竭、感染升级和远方信号迫使玩家移动，形成“建立联系—做出牺牲—带着后果上路”的节奏。

检验问题：长期龟缩是否永远优于冒险？如果是，迁徙主题没有进入规则。

## 5. 明确不做的内容

以下内容不进入 MVP，也不应被 AI 在实现过程中“顺手加入”：

- 无缝全球开放世界和逐街道长距离步行；
- 复杂车辆驾驶、射击弹道或动作连招；
- 大规模军团即时战略；
- 玩家对玩家、合作联机、排行榜和账号系统；
- 每个国家一套独立美术与剧情；
- 可自由搭建每一面墙的沙盒建造；
- 数百角色的人口模拟；
- 用生成式 AI 在玩家设备上实时写主线剧情；
- 对真实建筑内部进行一比一还原；
- 把现实灾害、真实个人住址或敏感设施作为猎奇卖点。

## 6. 统一术语

| 术语 | 定义 |
|---|---|
| 社区 | 玩家控制的全部幸存者、基地设施和共享资源的总称 |
| 基地 | 当前城市中社区驻扎和管理的具体地点 |
| 幸存者 | 可加入社区、拥有持久状态和关系的具名角色 |
| 小队 | 一次外出行动中最多 4 名幸存者组成的临时单位 |
| 世界层 | 展示城市、路线和远程信号的最高层地图 |
| 城市层 | 展示基地、道路、区域风险和兴趣点的行动规划地图 |
| 探索层 | 建筑内部、街口或特殊地点的战术网格场景 |
| 兴趣点（POI） | 可作为行动目标的建筑、设施或地标节点 |
| 行动 | 从出发到返回基地或转入下一地点的一次外出流程 |
| 事件 | 由条件触发、含选择和后果的叙事单元 |
| 事件链 | 共享状态并能跨时段延续的多个事件节点 |
| 导演 | 根据局势、节奏和冷却规则挑选事件与压力的系统 |
| 灰潮 | 世界内对 N7 灾难及感染浪潮的民间统称，名称暂定 |
| 警戒度 | 某区域因声音、尸体、火光等积累的短期威胁值 |
| 城市压力 | 随天数和关键行为增长的长期危险，防止无限搜刮 |
| 迁徙 | 放弃或封存当前基地，携带有限人员物资前往下一城市 |

## 7. 范围分层

| 系统 | 原型 | MVP | 正式版愿景 |
|---|---|---|---|
| 城市 | 手工灰盒街区 | 1 个精修城市 + 2 个测试区域 | 多气候地区包与任意城市稳定生成 |
| 战役 | 7 天验证循环 | 30 天、一次迁徙、结局变体 | 多城市连续远征与文明路线 |
| 幸存者 | 4 人、基础状态 | 4–12 人、关系、记忆、永久死亡 | 20 人、代际与跨城声望 |
| 基地 | 4 个设施 | 12 个设施、岗位、供电、防守 | 多基地、前哨和移动载具 |
| 敌人 | 1 种感染者 | 3 种感染者 + 敌对人类事件 | 地区生态、季节和势力战术 |
| 事件 | 10 个测试事件 | 60 个独立事件 + 10 条事件链 | 地区包、势力长线和创作者工具 |
| 地图数据 | 固定样例 | OSM 生成、缓存、回退和署名 | 自有服务、版本化地图包 |

## 8. 开工前待确认项

以下问题不影响当前策划完整性，但需要在对应里程碑前由创作者最终确认。表中给出默认方案，未确认时不得由 AI 擅自更换。

| 决策 | 当前默认 | 最晚确认阶段 | 影响 |
|---|---|---|---|
| 最终中文/英文名称 | 《末日坐标：Aftermap》暂定 | P4 前 | 商标检索、商店页、美术和本地化键 |
| 首个精修城市 | 选择创作者最熟悉且 OSM 数据完整的一座温带城市 | P2 结束前 | 地图样本、地区包、事件细节 |
| 引擎 | Godot 4.x + GDScript | P0 spike 结束 | 全部代码、工具和招聘/协作 |
| 首发语言 | 简体中文完整；英文先保证布局 | P4 前 | 文本生产、字体、测试和发行范围 |
| 首发区域与分级目标 | 根据实际发行地区确定，避免依赖高强度血腥表现 | P4 前 | 内容红线、商店问卷、隐私与法律文本 |
| 美术生产方式 | 占位像素素材验证后，再确定自制、外包或 AI 辅助整理 | P2 结束前 | 预算、版权、产能和风格统一 |
| 在线地图服务 | 开发期小范围合规接口；正式版使用自有或商业合规方案 | P3 结束前 | 成本、隐私、缓存、可用性和许可证 |
| 任意地点是否公开 | 仅在生成成功率和回退体验达到 Beta 标准后公开为实验功能 | P5 Beta | 宣传承诺、支持压力和地图测试量 |
| 正式版是否连续跨城 | MVP 数据接口保留，不在首版承诺 | MVP 反馈后 | 战役存档、内容量和世界层 UI |
| 现实地点内容保护规则 | 采用街区级隐私、现实机构中性叙事和人工审核 | P3 前 | 地图文本、分享功能、合规风险 |

确认时把结果移入“已冻结的设计决策”，并在变更记录中注明日期和影响范围。

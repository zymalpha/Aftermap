# 《末日坐标：Aftermap》技术架构、数据协议与 AI 协作

> 本文给出适合小团队和 AI 辅助的默认技术路线。正式开工前可用短原型更换引擎，但一旦进入纵向切片，就应冻结核心栈，避免 AI 反复重写。

## 1. 推荐技术栈

| 层 | 推荐 | 理由 |
|---|---|---|
| 游戏客户端 | Godot 4.x | 2D、TileMap、导航、UI 和跨平台能力适合本项目 |
| 游戏脚本 | 静态类型风格 GDScript | 代码短、迭代快、文本友好，便于 AI 阅读与审查 |
| 地图预处理 | Python 3.x 独立命令行工具 | GIS 数据解析、几何清理和批量测试生态更合适 |
| 内容数据 | JSON + JSON Schema | 人类和 AI 均易编辑，可自动校验和生成 |
| 本地化 | CSV/PO 风格表或 Godot 翻译资源 | 文本与逻辑分离 |
| 存档 | 版本化 JSON/二进制块 + 原子写入 | MVP 依赖少、易调试和迁移 |
| 版本控制 | Git + 大文件方案（素材增加后） | 追踪 AI 修改、回滚和审查 |
| 自动测试 | Godot headless + Python 测试 | 逻辑、数据和地图生成均可自动运行 |

不要在 MVP 同时引入后端账号、实时大模型、云存档、多人同步和复杂数据库。地图服务与游戏核心通过文件协议隔离，未来可替换为服务器而不重写模拟。

## 2. 架构原则

1. **场景不是数据真相**：角色、基地、地图和事件状态由 `GameSession` 持有，UI/场景只是展示；
2. **数据驱动**：物品、设施、角色特征、事件和模板由稳定 ID 与数据定义，不散落在脚本常量；
3. **命令改变状态**：UI 发送命令，领域系统校验并返回结果，UI 不直接改库存或生命；
4. **随机可复现**：所有影响玩法的随机数来自命名随机流并存档；
5. **模块单向依赖**：模拟不依赖 UI，内容不依赖具体场景，地图工具不依赖玩家存档；
6. **失败可恢复**：存档、地图生成、场景切换均有校验和恢复点；
7. **先可测后扩展**：每个系统先提供无画面测试接口，再连接表现。

## 3. 模块关系

```text
UI / 输入 / 音频 / 视觉表现
              ↓ 命令 / 查询
应用层：GameSession、流程状态机、场景协调
              ↓
领域层：时间、角色、关系、基地、库存、战斗、感染、事件、目标
              ↓
内容层：ContentDB、模板、数值、文本键、Schema

外部适配器：存档、地图包、地图下载、日志、本地化
              ↑ 只通过明确接口接入应用层
```

禁止 UI 节点直接读取地图下载器，或事件脚本直接查找场景树中的角色节点。

## 5. 核心状态模型

`GameSession` 是当前战役唯一根状态，建议包含：

```text
metadata          存档版本、内容版本、种子、创建时间
campaign          天数、幕、难度、结局状态
clock             城市日期、分钟、时段
rng_streams       各命名随机流状态
community         成员 ID、共享资源、承诺、社区记忆
survivors         角色状态字典
relationships     有向关系记录
base              当前基地、设施、岗位、队列
world             城市、分区、POI、路线、势力、压力
tactical_session  当前探索场景状态（若有）
events            事件链、旗标、延迟计划、冷却
objectives        目标与承诺状态
inventory         容器、物品实例、保留线
history           关键日志和结局摘要
```

实体之间只保存稳定 ID，不保存运行时节点引用。

## 6. ID 和实例

### 6.1 定义 ID

定义 ID 指内容类型，如 `wpn_pistol_9mm`。规则：

- 小写 ASCII、数字、下划线；
- 带类型前缀；
- 发布后不复用已删除 ID；
- 显示名称始终来自本地化键；
- 重命名通过迁移表兼容旧存档。

### 6.2 实例 ID

实例 ID 指某个具体对象，如一把有耐久的手枪。使用战役内唯一、不可猜测顺序不重要的 ID，例如 UUID 或“种子命名空间 + 递增序号”。

角色、物品、伤口、记忆、POI 和事件实例均需要实例 ID。日志引用实例 ID，显示时再查当前/历史名称。

## 7. 内容加载

启动时：

1. 读取 schema 版本；
2. 加载核心内容清单；
3. 校验文件结构和引用；
4. 检查重复 ID、缺失文本键、无效标签和循环依赖；
5. 建立只读 `ContentDB`；
6. 计算内容指纹；
7. 加载存档时比较版本并执行迁移。

正式构建遇到单条非关键内容错误时应隔离该内容并记录；核心物品、主线或 schema 错误则阻止开始并显示可理解错误。开发构建直接失败，避免坏数据继续传播。

## 9. 命令与领域事件

### 9.1 命令

命令代表意图，例如：

- `AssignJob`；
- `PrepareExpedition`；
- `MoveTacticalUnit`；
- `UseItem`；
- `StartFacilityUpgrade`；
- `ChooseEventOption`；
- `EndDay`；
- `PackMigrationCargo`。

命令先校验，成功后一次性修改状态并发布领域事件。失败不能部分扣资源。

### 9.2 领域事件

领域事件是已经发生的事实：

- `ResourceConsumed`；
- `SurvivorWounded`；
- `RelationshipChanged`；
- `FacilityPowerLost`；
- `PoiAlertChanged`；
- `PromiseBroken`。

日志、UI、音频和事件导演可以订阅，但不应互相循环触发无限变化。领域事件在单个命令事务结束后按稳定顺序派发。

## 10. 游戏流程状态机

顶层状态：

```text
Boot
→ MainMenu
→ MapSetup
→ CampaignLoad
→ MorningReport
→ BasePlanning
→ CityPlanning
→ TacticalExploration
→ ReturnResolution
→ NightManagement
→ NightResolution
→ Ending
```

每个状态明确：允许的命令、进入数据、退出条件、存档点和失败回退。场景切换不能靠多个 UI 各自猜测当前阶段。

## 11. 时间模拟

- 战略工作使用整数分钟和离散结算；
- 战术移动使用固定物理步长，默认 60 Hz；
- 战术长动作累计整数毫秒或固定 tick，完成后换算城市分钟；
- 暂停时不推进任何领域时钟；
- 2 倍速只改变处理 tick 数，不改变随机调用顺序；
- 夜间结算按 `03` 的固定顺序执行；
- 快进时仍运行相同规则，不调用另一套简化公式。

## 12. 随机数设计

不要使用全局随机函数。`RngService` 提供命名流：

- `world_generation`；
- `city_state`；
- `poi_scene_<poi_id>`；
- `daily_director_<day>`；
- `event_<event_instance_id>`；
- `combat_<tactical_session_id>`；
- `character_generation`；
- `cosmetic_only`。

玩法随机流保存内部状态或可重建计数。纯视觉粒子使用独立流，不能因多生成一滴雨改变下一次命中或事件真相。

确定性测试用同一命令序列运行两次并比较领域状态哈希。

## 13. 战术架构

### 13.1 模拟与表现分离

- `TacticalSimulation` 持有格子、单位、声音、警觉、动作和伤害；
- Godot 节点读取模拟快照并播放动画；
- 动画完成不能决定攻击是否命中，只确认表现；
- 单位行为使用可测试状态机，不在 `_process` 中散落决策；
- 路径请求有唯一请求 ID，过期结果不得覆盖新命令。

### 13.2 行为状态

角色：`idle / moving / interacting / aiming / attacking / reloading / treating / grabbed / downed / evacuating`。

感染者：`idle / wandering / suspicious / investigating / alerted / chasing / attacking / staggered / downed / dead`。

每次转换记录原因，调试器可查看最近 20 次转换。

### 13.3 声音

声音系统接收离散 `SoundPulse`，在导航/房间图传播后通知感知者。不要为每个单位每帧执行全地图半径搜索。短时间相近声源可合并，但枪声和关键诱饵不能被吞。

## 14. 事件解释器

- 条件和效果类型采用注册表白名单；
- 数据不能携带脚本路径、表达式求值或系统命令；
- 每个效果先做 dry-run，确认实体和资源合法；
- 整组选项效果作为事务应用；
- 失败时回滚并写结构化错误；
- 延迟事件使用绝对战役日/分钟和稳定实例 ID；
- 调试器可输出条件树、评分和执行差异。

事件解释器的单元测试优先级高于事件数量生产。

## 15. 地图工具边界

Python 地图流水线只输入 OSM 快照和生成参数，只输出版本化地图包，不直接写玩家存档。

### 15.1 输入

```text
confirmed_location.json
osm_snapshot.*
pipeline_config.json
content_taxonomy.json
seed
```

### 15.2 输出

```text
manifest.json
city_graph.json / compact binary
districts.json
pois.json
geometry.*
generation_report.json
attribution.txt
```

### 15.3 CLI 参考

```text
map_pipeline fetch --location <file> --out <snapshot_dir>
map_pipeline build --snapshot <dir> --config <file> --out <map_pack>
map_pipeline validate --map-pack <dir> --report <file>
map_pipeline preview --map-pack <dir> --out <png>
```

实际 CLI 名称可以调整，但 fetch、build、validate 必须能独立运行，方便缓存、测试和 AI 排错。

## 16. 存档

### 16.1 文件结构

建议每个槽包含：

```text
slot_<id>/
├─ header.json          # 标题页所需最小信息
├─ session.dat          # 主状态，压缩或二进制块
├─ tactical.dat         # 当前场景状态，可选
├─ history.jsonl        # 关键战役日志
├─ screenshot.png       # 可选预览
└─ checksums.json
```

### 16.2 原子写入

1. 写入同目录临时文件；
2. 刷新并校验；
3. 保留上一版为恢复副本；
4. 原子替换正式文件；
5. 更新 header；
6. 失败时不删除最后可用存档。

### 16.3 版本迁移

- 存档包含 `save_schema_version` 和 `content_fingerprint`；
- 每个版本只负责迁移到下一版本，链式执行；
- 迁移前自动备份；
- 已删除内容使用占位/映射，不让整个存档崩溃；
- 迁移测试保留至少 3 个历史版本夹具；
- 开发构建可导出可读 JSON 快照帮助排错。

## 17. 地图包与存档关系

- 存档引用地图包 ID、manifest 指纹和不可变基础数据；
- 动态 POI 状态保存在存档，不回写共享地图包；
- 删除地图包前查询所有依赖存档；
- 地图包升级生成新 ID，旧战役继续使用旧包；
- 地图缺失时先尝试恢复/重新取得相同快照，不能悄悄换成新地图；
- 预设地图随构建版本提供并可校验完整性。

## 18. 本地化架构

- 数据只存 `name_key`、`description_key` 等键；
- 插值参数使用具名字段，如 `{survivor_name}`；
- 不把多个可变词片段拼成中文/英文句子；
- 复数、性别和语序由本地化模板处理；
- 启动校验缺失键、无用键和占位符不一致；
- 地图真实名称与游戏文本分开，允许原名 + 转写；
- 事件文本版本与逻辑版本分别记录，纯文字修正不应破坏存档。

## 19. 日志与错误

结构化日志字段：时间、构建、存档槽、系统、严重度、事件 ID、实体 ID、错误码。日志禁止写入精确玩家输入地址、完整外部响应或个人信息。

严重度：

- `TRACE`：开发状态转换；
- `INFO`：主要流程和内容加载；
- `WARN`：可回退的数据问题；
- `ERROR`：当前操作失败但游戏可继续；
- `FATAL`：无法保证状态一致，停止并保留恢复点。

玩家看到本地化说明和错误编号，不看到内部堆栈；开发日志保留堆栈。

## 20. 安全与输入防护

- 地点和存档名不能直接成为文件路径；
- 所有外部文件路径规范化并限制在指定目录；
- 地图包和内容包检查大小、schema、校验和与压缩炸弹；
- JSON 事件不执行脚本、表达式或网络请求；
- 外部地图文本按不可信输入处理并转义显示；
- 网络请求设超时、大小上限、速率限制和取消；
- API 密钥不写入仓库或客户端明文配置；
- 崩溃报告默认移除坐标、用户名和文件绝对路径；
- 模组/自定义内容若后置实现，使用独立目录和权限边界。

## 21. 性能预算

首轮目标硬件以常见集成显卡/中低端独显 PC 为参考，实际最低配置由测试确定。

| 项目 | 目标 |
|---|---|
| 战术帧率 | 1080p 下 60 fps，压力场景最低 30 fps |
| 活跃单位 | 4 名角色 + 30 个模拟 AI，上屏推荐远低于此值 |
| 普通存档 | 2 秒内完成，不阻塞到无响应 |
| 普通读取 | 5 秒内进入可操作界面 |
| 场景切换 | 目标 5 秒内，显示可取消/可恢复状态 |
| 城市 POI | 120 个可行动节点保持地图交互流畅 |
| 城市初次生成 | 常规数据 60 秒内预览，超时可取消 |
| 内存 | MVP 运行目标低于 2 GB，最终按素材测试调整 |
| 自动结算 | 12 人、30 天状态下夜间结算低于 200 ms 逻辑时间 |

先测量再优化。禁止仅凭猜测引入对象池、并行任务或二进制格式。

## 22. 测试分层

### 22.1 单元测试

- 消耗、负重、技能、感染公式；
- 库存事务和保留线；
- 关系与记忆变化；
- 事件条件/效果；
- 随机流确定性；
- 存档迁移；
- 路线成本和工作队列。

### 22.2 集成测试

- 晨间到夜间完整一天；
- 外出、搜刮、返回、治疗和消耗；
- 角色死亡对岗位、关系、事件和结局的更新；
- 断电导致设施暂停并恢复；
- 事件链跨天和关键角色死亡替代；
- 地图包加载、缺失和恢复；
- 战术中保存、读取和撤退。

### 22.3 模拟测试

- 100–1000 个固定种子自动跑 30 天简化策略；
- 检查无资源死局、状态越界、事件无出口和人口异常；
- 比较相同命令序列状态哈希；
- 地图生成对高密、低密、河流和数据稀疏样本做模糊测试。

### 22.4 表现测试

- 多分辨率截图差异；
- 160% UI 与最长文本；
- 色觉模拟与仅键盘操作；
- 30 AI、雨、火、光照叠加压力场景；
- 音频优先级和字幕同步。

## 23. 构建与持续集成

每次合并至少运行：

1. GDScript 静态检查；
2. JSON Schema 与引用校验；
3. 单元测试；
4. 关键一天循环集成测试；
5. 固定种子确定性测试；
6. 地图工具测试；
7. 无界面启动和加载预设地图；
8. 构建产物版本与内容指纹写入。

夜间任务再运行地图样本矩阵、100 局模拟和截图测试。测试失败不得让 AI 用删除断言或扩大容差掩盖，必须解释是实现错误还是基线变化。

## 24. 架构决策记录

重大选择写入 `docs/adr/ADR-XXXX-标题.md`：

```text
状态：提议/接受/废弃/替代
背景：为什么需要决定
决策：选择了什么
备选：考虑过什么
后果：获得与失去什么
影响模块：
迁移计划：
```

至少记录：引擎选择、存档格式、随机确定性、地图数据服务、事件解释器、模组边界。

## 25. AI 修改代码的边界

每个 AI 任务只应有一个主要交付物和一组可运行验收。AI 必须：

- 先读相关策划、接口和现有测试；
- 检查工作区现有改动，不覆盖未知修改；
- 沿用当前模块和命名；
- 先写/更新测试，再实现或同步实现；
- 不添加未批准依赖、全局单例或远程服务；
- 不改无关格式和文件；
- 运行限定测试并报告未运行项；
- 对规则歧义记录假设，不悄悄发明新玩法；
- 更新对应文档或 ADR，若接口/规则确实改变。

禁止一次性要求 AI “做完整游戏”。单个任务最好在 1–6 个紧密相关文件内完成，跨模块工作先拆接口任务。

## 26. AI 代码审查清单

- 是否把 UI 当成了状态源？
- 是否使用了全局随机数？
- 是否使用显示名称代替稳定 ID？
- 失败时是否产生了部分扣费/部分状态？
- 是否在事件数据中加入任意脚本执行？
- 是否让读取存档重新抽取结果？
- 是否新增依赖但没有 ADR 和许可证检查？
- 是否覆盖用户/其他任务的现有改动？
- 是否只测试了快乐路径？
- 是否处理角色死亡、缺数据、断网、取消和旧存档？
- 是否把可访问性和本地化文本写死？
- 是否在日志中泄露坐标、路径或 API 密钥？

## 27. 开工前技术验证

正式纵向切片前完成 6 个独立 spike：

1. 4 角色网格移动、暂停命令和队列；
2. 30 个声音感知单位的性能与行为；
3. JSON 事件条件/效果事务和调试输出；
4. 一个真实 OSM 小区域转城市道路图和预览；
5. 原子存档、故意中断与恢复；
6. 32×32 像素素材在多分辨率下稳定渲染。

每个 spike 只回答技术问题，可丢弃其代码。通过标准写入 ADR，再进入生产架构。

# 《末日坐标：Aftermap》AI 任务拆分与提示词模板

## 1. 目的

这份文档用于后续让 AI 持续参与代码、内容、测试和素材制作。核心原则不是写一条“帮我完成游戏”的超长提示，而是让每次任务都有确定上下文、明确边界、可运行验收和可追踪交接。

AI 最适合：

- 按既有接口实现一个小模块；
- 为规则补单元测试和边界测试；
- 生成符合 schema 的内容初稿；
- 制作调试工具、数据校验器和批处理脚本；
- 分析日志、复现缺陷和提出有限修复；
- 按固定尺寸生产占位或批量素材变体；
- 检查文档、数据和代码是否一致。

AI 不应独立决定产品方向、替代真人试玩、无审核写入正式剧情，或在没有迁移方案时大规模重构。

## 2. 开发开始时新增的生产文档

建立实际工程后，在 `docs/production/` 维护四份短文档：

### `PROJECT_STATE.md`

只写当前事实：

```text
当前里程碑：P1 战术实验室
当前可运行构建：0.1.3-dev
已完成：TAC-001、TAC-002
进行中：TAC-003
当前阻塞：门打开后导航网格刷新不稳定
下一验收：灰盒诊所从入口到出口可达
已知重要缺陷：BUG-0012
最后更新：日期 / 提交
```

### `BACKLOG.md`

每个任务一行状态和链接，不在聊天记录里保存唯一待办。

### `DECISIONS.md`

记录尚未达到 ADR 规模但会影响多个任务的短决定，如“搜索进度中断后保留 50%”。

### `CHANGELOG_DEV.md`

按构建记录玩家可感知变化、存档兼容和测试注意事项。

每次新 AI 会话先读 `PROJECT_STATE.md`，结束前只在实际状态变化时更新它。

## 3. 上下文加载顺序

不要把全部文档无差别塞给每个 AI。按任务读取：

1. `00_文档导航与决策总表.md` 中相关冻结决策；
2. 当前任务对应专项策划；
3. `11_技术架构数据协议与AI协作.md` 的相关接口规则；
4. `PROJECT_STATE.md` 和任务卡；
5. 目标模块代码、测试和相邻接口；
6. 只有出现引用时再读取其他文档。

| 任务类型 | 必读策划 |
|---|---|
| 战术移动/战斗 | 03、04、06、09、10、11 |
| 角色/关系 | 02、04、08、09、11 |
| 基地/资源 | 03、05、09、10、11 |
| 地图 | 01、02、07、10、11 |
| 事件 | 02、04、08、09、11 |
| UI | 对应系统文档 + 10、11 |
| 存档/随机 | 03、08、11 |
| 里程碑/测试 | 12 + 对应系统文档 |

## 4. 合格任务卡

一张任务卡必须包含：

| 字段 | 要求 |
|---|---|
| ID | 稳定且唯一，如 `TAC-004B` |
| 目标 | 一句话描述完成后的可见结果 |
| 原因 | 它服务哪个玩家体验或里程碑 |
| 范围内 | 明确要改什么 |
| 范围外 | 明确这次不能顺手做什么 |
| 输入 | 相关文件、接口、数据和固定种子 |
| 规则 | 必须遵守的策划条款 |
| 验收 | 可运行、可观察、最好自动化 |
| 边界 | 至少列出失败、取消、空数据、旧状态之一 |
| 交付 | 代码、测试、数据、文档或报告 |
| 依赖 | 前置任务与允许调用的接口 |
| 风险 | 可能影响存档、性能或其他模块的点 |

如果任务卡无法在 5 分钟内说清“完成时如何证明”，它还不够小。

## 5. 任务大小

推荐单任务规模：

- 1 个主要行为；
- 1–6 个紧密相关生产文件；
- 同轮测试可在数分钟内运行；
- 一次提交可完整说明；
- 不同时改领域规则、UI 设计、存档 schema 和美术管线。

需要拆分的信号：

- 标题含“以及”“完整系统”“全部”；
- 验收超过 8 条且属于不同层；
- 需要新依赖、新 schema 和新 UI；
- 预计同时触碰 3 个以上领域模块；
- 无法在失败时整体回退；
- AI 必须自行发明接口才能继续。

## 6. 大任务拆分示例

错误任务：

> 实现完整战斗、受伤、感染和救援系统。

合理拆分：

1. `TAC-006A`：纯数据的伤害结算与生命阈值；
2. `TAC-006B`：伤口生成、部位和出血 tick；
3. `TAC-006C`：倒地状态和稳定窗口；
4. `TAC-006D`：队友稳定动作及中断；
5. `TAC-006E`：拖拽/背负与负重；
6. `TAC-006F`：撤离时倒地者结算；
7. `INF-001A`：暴露剂量和防护；
8. `INF-001B`：清创与建立前变化；
9. `INF-001C`：夜间感染阶段增长；
10. `UI-TAC-004`：状态显示和因果日志。

每一步先有领域测试，再接动画和 HUD。

## 8. 完整代码任务示例

```text
你正在维护《末日坐标：Aftermap》项目。

任务 ID：CORE-004A
目标：实现夜间结算的资源消耗步骤，使 4 名角色按口粮政策原子扣除食物和净水，并生成可追溯领域事件。

开始前必须阅读：
- 策划案/03_核心循环与战役流程.md 第 9 节
- 策划案/05_基地资源与制造系统.md 第 7 节
- 策划案/09_数值基线与内容清单.md 第 3、5 节
- 策划案/11_技术架构数据协议与AI协作.md 第 5、9、16 节
- game/domain/inventory 下现有代码和测试

范围内：
- 正常、节省、优先康复三种政策的资源需求计算
- 资源充足与不足时的确定性分配
- ResourceConsumed、RationShortage 领域事件
- 幂等或防止同一夜重复结算的保护

范围外：
- 不实现精力恢复、关系事件、感染增长和 UI
- 不改变资源单位和角色状态字段
- 不新增全局单例或依赖

必须遵守：
- 健康成人基线为 2 口粮、3 净水/天
- 事务失败不能部分扣除后不记录
- 所有变化可由日志解释
- 相同状态和命令得到相同结果

验收：
1. 4 人正常政策从 20 口粮/18 水变为 12/6；
2. 缺水时按已定义优先级分配并为未满足角色生成结果；
3. 死亡/离队者不参与消耗；
4. 同一夜重复调用不会二次扣除；
5. 单元测试和一天循环测试通过。

先阅读再改代码，完成后报告实际命令输出和未覆盖风险。
```

```text

1. ...
2. ...

可能相关文件：<不要断言根因>

要求：
- 先复现并新增失败测试；
- 找到最小根因，不通过吞异常、延时或扩大容差掩盖；
- 不改变策划规则来适配错误实现；
- 检查相同根因是否影响存档/事件/其他场景；
- 修复后运行回归测试并提供结果；
- 若无法复现，保留证据并说明还缺什么，不猜测提交修复。
```

```text
任务 ID：CONTENT-EVT-<编号>
请为《末日坐标：Aftermap》起草 <数量> 个结构化事件。

必读：
- 02_世界观与叙事设计.md

- 09_数值基线与内容清单.md
- 当前 event schema、标签白名单、效果白名单

内容缺口：<主题与配额>
战役阶段：<天数/幕>
目标强度：<0-4>
地点：<允许 POI>
要求使用的角色标签：<背景/性格/关系>
禁止内容：<红线与本批特殊限制>

每个事件先输出策划模板：
- 一句话冲突
- 为什么现在发生
- 已知信息/隐藏真相
- 2-5 个行动选项
- 立即与延迟效果
- 角色意见
- 超时、拒绝、关键角色死亡回退
- 结局摘要

经结构确认后，再输出符合 schema 的 JSON 和本地化文本。只能使用给定 ID、条件、效果和文本键规则；不确定的 ID 标为待创建，不能自行假造已经存在。
```

### 事件人工审查问题

- 是否只有一个显然正确且无代价的选项？
- 风险是否可从情境判断？
- 是否违反感染传播规则？
- 是否把职业/国家/疾病写成刻板善恶？
- 是否真的引用当前角色和地点，而非只替换名字？
- 延迟后果能否回溯原决定？
- 关键角色死亡或玩家忽略时能否结束？
- 是否至少改变一个持久状态？
- 文本是否可以更短、更具体？

## 11. 角色内容生成模板

```text
生成 <数量> 个幸存者候选数据，不写固定主角剧情。

使用：现有背景、性格、底线、弱点和专长 ID 白名单。
约束：
- 五项技能 0-5，遵守背景倾向；
- 每人 1 背景、2 性格、1 底线、1 弱点、最多 1 初始专长；
- 性格不能只是职业刻板印象；
- 给出一件合理随身物和一个可生成个人目标；
- 给出能与另一候选建立的初始记忆钩子；
- 不生成儿童、真实个人或现实私人地址；
- 输出 JSON 数据和 80 字以内玩家简介；
- 通过既有 schema，不创建未批准 ID。

同时给出组合检查：医疗、工程、战斗、搜索/社交覆盖，重复组合和必然冲突。
```

AI 可以提供候选，但初始 4 人组合必须由程序约束验证，不能只相信文本说明。

## 17. 代码审查提示词

```text
以缺陷审查方式检查 <提交/diff>，不要先总结优点。

必读：相关策划、11_技术架构数据协议与AI协作.md、任务卡验收。
优先寻找：
- 行为回归、状态不一致、存档兼容；
- 非确定随机、读档重抽、事件事务不完整；
- UI 直接改领域状态；
- ID/文本/数据引用错误；
- 角色死亡、取消、断网和坏数据回退；
- 性能数量级问题；
- 隐私、安全、许可证和日志泄露；
- 测试遗漏或只测试实现细节。

输出按严重度排序，每条包含文件与行号、触发条件、具体后果和可验证修复方向。若没有问题，明确说明剩余测试空白和风险。
```

## 18. AI 输出契约

每次任务结束报告固定四项：

```text
完成：实际改变了什么玩家/系统行为
文件：新增或修改的核心文件
验证：运行的命令与结果；未运行项及原因
风险：假设、兼容性、仍待处理问题
```

禁止只说“已完成”“应该可以”。测试输出要给关键结论；无法运行时必须明确。

## 20. 多任务并行规则

即使使用多个 AI，也只并行边界明确、文件不重叠的任务：

- 可以并行：事件文本初稿与战术单元测试；
- 可以并行：地图样本验证与角色头像概念；
- 不宜并行：两个任务同时改 `GameSession`；
- 不宜并行：schema 尚未冻结时批量生成内容；
- 不宜并行：存档格式和实体 ID 同时重构；
- 不宜并行：同一个 UI 场景的布局与领域接口一起由不同任务修改。

每个并行任务使用独立分支/工作树和任务 ID。合并前先更新主线、运行全套相关测试，再解决语义冲突，不能只处理文本冲突。

## 21. 防止 AI 自行扩张

提示词中明确加入：

- “范围外功能只记录建议，不实现”；
- “不得添加依赖，除非任务卡明确批准”；
- “不得创建重复管理器或全局单例”；
- “不得为了测试通过修改策划常量”；
- “不改无关文件格式”；
- “不使用实时生成式 AI 作为游戏运行依赖”；
- “遇到冲突先给证据和最小阻塞，不自行重写整个模块”。

审查 diff 时出现以下信号应暂停合并：

- 任务外大量文件被格式化；
- 新增 `Manager2`、`NewSystem` 等临时重复模块；
- 测试被删除或断言放宽却无说明；
- 用硬编码名称绕过 ID 系统；
- 异常被空捕获；
- 为方便直接暴露可写全局字典；
- 引入网络请求或密钥；
- 存档字段变化但无版本迁移；
- 数据 schema 与示例没有同步。

## 22. 内容批量生产顺序

不要一开始让 AI 生成全部 60 个事件。正确顺序：

1. 手工确定 schema、标签和效果白名单；
2. 生成 3 个不同主题事件；
3. 通过结构、文本、状态和试玩审查；
4. 修改模板与校验器；
5. 批量生成 10 个；
6. 运行 ID、引用、可达性和重复检测；
7. 人工润色并试玩；
8. 按内容缺口继续下一批；
9. 最后做全库一致性和节奏审查。

角色、物品、配方、POI 模板和素材也使用“少量样本 → 验收 → 批量”的同样流程。

## 24. 每轮 AI 工作的完成定义

一轮任务只有在以下条件都满足时完成：

- 行为符合任务卡和策划；
- 修改范围没有悄悄扩大；
- 相关测试通过，并记录实际结果；
- 错误、取消和至少一个边界路径有覆盖；
- 数据、ID、文本和存档兼容已检查；
- diff 中没有未知覆盖或无关格式化；
- 必要文档/ADR/生产状态已同步；
- 下一任务可以只依赖仓库事实继续，不依赖聊天记忆。

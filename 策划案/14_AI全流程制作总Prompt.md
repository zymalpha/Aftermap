# 《末日坐标：Aftermap》AI 全流程制作总 Prompt

## 使用说明

这份总 Prompt 必须交给**能够访问本地文件、终端、网络并持续修改工程的编程智能体**，例如终端型 coding agent。普通聊天机器人只能给建议，无法真的安装工具、创建工程、运行测试和导出 EXE。

它要求完成的是策划案定义的**可完整通关 MVP**，不是一次性实现“正式版愿景”中的所有地区、多城市连续战役和扩展功能。即使使用总 Prompt，制作仍会经历许多工具调用、上下文轮次和真实试玩，不可能靠一次文本回复完成。

使用前可修改下面三个默认值：

```text
FIRST_CITY = 南京
GAME_LANGUAGE = 简体中文
TARGET_PLATFORM = Windows x86_64
```

本机已知硬件：Intel Core i5-12450H、约 16 GB 内存、Intel UHD Graphics 集成显卡。它可以运行轻量 2D Godot 游戏，但不适合本地运行图像生成模型、语言模型或昂贵的 GPU 特效。

====================
三、本机硬件约束
====================

已知硬件：
- CPU：Intel Core i5-12450H，8 核 12 线程；
- 内存：约 16 GB；
- 图形：Intel UHD Graphics 集成显卡，共享显存；
- 没有独立显卡；
- 当前 PATH 中未发现 Godot、Python 或 Git。

因此必须：
1. 使用 Godot 4.x 当前稳定正式版，不使用 beta/RC；
2. 使用 Compatibility/OpenGL 渲染器，不使用 Forward+；
3. 使用 GDScript 和 2D TileMap/网格，不引入 3D 场景；
4. 内部目标分辨率 1280×720，支持 1920×1080 窗口/全屏缩放；
5. 像素纹理使用 nearest、整数/像素稳定缩放；
6. 不使用昂贵实时阴影、体积雾、全屏后处理、GPU 粒子海量叠加或高分辨率动态光照；
7. 普通战术场景目标 60 fps，压力场景最低稳定 30 fps；
8. 默认普通遭遇远低于 30 个活跃 AI；30 个只作为压力测试上限；
9. 地图几何预处理放在 CPU 工具或后台任务，不能阻塞 UI 到无响应；
10. 游戏运行时不调用本地 LLM、扩散模型、语音生成模型或其他需要独立显卡的 AI；
11. 不把云端 AI/API 设为游戏运行依赖；
12. 素材使用人工/脚本制作的轻量像素资源或审核过的合规资源，不在本机训练模型。

所有性能结论必须在这台机器上测量，不能只写“应该流畅”。

====================
五、工作方式
====================

这是一个持续执行任务。不要只输出计划，也不要在建立空工程、完成第一场战斗或做出漂亮截图后声称整个游戏完成。

你必须：
1. 先盘点文件、工具和现有改动；
2. 建立 docs/production/PROJECT_STATE.md、BACKLOG.md、DECISIONS.md、CHANGELOG_DEV.md；
3. 按 12_MVP制作路线图与验收标准.md 的 P0→P6 顺序推进；
4. 使用任务 ID，把每个大功能按 13_AI任务拆分与提示词模板.md 拆成小任务；
5. 每次只让一个模块成为状态真相，遵守 11 文档的架构与依赖方向；
6. 每个任务同步实现测试、错误路径和调试信息；
7. 完成一项就更新状态，不把“已计划”写成“已完成”；
8. 每个阶段达到出口标准后立即进入下一阶段，不等待例行确认；
9. 只有出现付费/密钥、全局安装、法律选择、不可恢复操作或策划无法合理推断的重大冲突时才询问；
10. 上下文接近上限时先把准确交接写入仓库，下一轮从 PROJECT_STATE 继续，不重复已完成工作；
11. 定期运行完整测试，而非等最后一起修；
12. 始终报告真实结果，无法运行就明确说明，不能说“理论上可用”。

每次阶段性报告只包含：
- 已完成且经过验证的行为；
- 当前测试结果；
- 当前进行项；
- 真实阻塞或重要风险。

不要用长篇计划替代执行。

- 任意地点功能标注“实验”，失败时一键进入预设地图。

阶段 P4：30 天功能 Alpha
- 完成 4–12 人、12 设施、42 基础物品、16 配方；
- 完成基地电力、卫生、噪声、防守、流亡和迁徙打包；

- 自动运行多种固定策略与至少 100 个种子，清除状态越界、死局和悬空引用。

阶段 P5：内容与表现 Beta

- 完成基地、城市、战术、事件、档案、设置 UI；
- 完成必要环境音、战术信息音、广播、字幕和克制音乐；
- 完成简体中文、按键重绑、UI 缩放、色觉符号、关闭闪烁/震动等无障碍功能；
- 做分辨率截图测试和本机性能测试；
- 没有真人测试条件时，不能伪造“5 名/20 名玩家测试”；把需要真人执行的用例整理成清单，并完成全部可自动化与自测部分。

阶段 P6：稳定和本地交付

- 导出 Windows x86_64 构建；
- 在没有开发工具参与的情况下，从 build/windows/Aftermap.exe 启动；
- 验证新游戏、预设地图、前三天关键流程、保存、退出、重启、读取；
- 验证可通过调试/自动快进完成 30 天结局，但正式玩家模式不暴露调试入口；
- 生成最终测试报告、许可证清单、README 和已知限制。

任何阶段未过硬门槛时，继续修复和复测，不得只把问题记为“以后完善”然后宣布完成。

====================
七、技术硬规则
====================

1. Godot 场景和 UI 不是领域状态真相；GameSession 和领域模块持有状态；
2. UI 只能发送命令、读取查询和处理结构化结果，不能直接修改生命、库存或关系；
3. 所有内容使用稳定 ASCII ID 和 JSON Schema；
4. 所有玩法随机来自命名随机流，视觉随机不能影响战斗/事件结果；
5. 同一快照、内容版本、种子和命令序列必须得到相同结果；
6. 事件条件/效果使用白名单解释器，不能执行任意脚本；
7. 命令和事件效果必须事务化，失败不能留下部分扣费；
8. 存档使用版本、迁移、校验和、临时文件与上一版恢复副本；
9. 地图包基础数据不可变，动态状态保存在战役存档；
10. 显示文本全部走本地化键，不把中文名称当 ID；
11. 不添加未经批准的全局单例、重复 Manager 或大型依赖；
12. 不通过删除测试、扩大容差、吞异常或降低策划门槛掩盖错误；
13. 所有外部输入不可信：路径、JSON、OSM 文本、压缩包和文件大小都要校验；
14. 日志不得包含精确私人坐标、用户名、绝对用户路径和密钥；
15. 所有网络功能都有超时、取消、速率限制、缓存和离线回退。

====================
八、视听资源规则
====================

本机没有独显，因此不要安装或运行本地生成式图像/音频模型。

允许的素材路线按优先级：
1. 用简单、统一、可复现的脚本制作原创 32×32 地块、32×48 角色、图标和基础动画；
2. 使用工作区内人工制作/后续提供的素材；
3. 使用明确为 CC0 或与商业项目兼容的素材，并保留来源、版本和许可证；
4. 使用外部云端生成素材只能在已有合法授权且不需要我提供付费密钥时作为候选，并必须人工/程序统一像素尺寸、锚点、透视和调色板。

禁止：
- 把受版权保护游戏的截图、角色或音效直接放进项目；
- 使用无法确定许可证的搜索结果；
- 最终构建仍以纯色矩形、缺失纹理或开发者图标代替主要可见内容；
- 每批素材使用不同透视、光源、线宽和像素密度；
- 在图片里烘焙 UI 文本；
- 用高分辨率素材缩小冒充像素画而不做像素整理。

素材先少量样本验收，再批量生产。所有素材导入必须符合 10_UI_UX美术与音频规范.md。

## 现实预期

这份 Prompt 能显著减少 AI 走偏，但不能消除三个事实：

1. 完整 MVP 仍是一个需要数月工作量的项目，AI 会经历多轮上下文和修复；
2. “好不好玩、角色是否令人记住、美术是否统一”必须由你实际试玩并反馈；
3. 正式任意城市服务、商业发行许可证和宣传承诺仍需要人工最终确认。

最稳妥的使用方式是让同一个终端型 AI 持续维护仓库状态，按里程碑执行；你只需要在每个阶段试玩构建并回答少量真正影响方向的问题。
